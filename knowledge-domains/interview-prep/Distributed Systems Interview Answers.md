# Distributed Systems Interview Answers

分布式系统题考的是故障视角：网络会抖、节点会挂、消息会重复、数据会延迟、时钟会漂移、调用会超时。

## CAP 怎么理解？

参考回答：

CAP 讨论的是分布式系统发生网络分区时，一致性和可用性之间必须取舍。因为网络分区无法完全避免，所以真正的选择是故障时更偏 CP 还是 AP。

强交易系统通常偏一致性，宁愿拒绝写也不能写错；评论数、点赞数、推荐特征这类可以偏可用和最终一致。不要脱离业务场景背 CAP。

## BASE 和最终一致是什么？

参考回答：

BASE 是 Basically Available、Soft state、Eventually consistent。它承认分布式系统里很多场景没必要强一致，可以先保证基本可用，允许中间状态，最终通过重试、补偿、异步同步达到一致。

最终一致不是“不管一致性”，而是要有状态机、重试、对账、DLQ、补偿和监控来保证最终能收敛。

## RPC 和 REST 怎么选？

参考回答：

REST/HTTP 通用、可读、跨语言友好，适合对外 API 和资源模型。RPC 更像方法调用，通常有 IDL、代码生成、连接复用和更强类型，适合内部服务高频调用。

无论选哪个，远程调用都不是本地调用。必须处理超时、重试、熔断、限流、幂等和版本兼容。

## 服务注册和服务发现解决什么问题？

参考回答：

服务实例会扩缩容、重启、迁移，调用方不能写死 IP。注册中心保存服务实例列表，服务发现让调用方动态获取可用实例，并结合健康检查和负载均衡。

关键问题是注册信息一致性、健康检查准确性、缓存过期、故障实例摘除和雪崩时的保护。

## 分布式 ID 怎么设计？

参考回答：

常见方案包括数据库自增、号段、UUID、Snowflake、Redis 计数器。数据库自增简单但扩展性差；UUID 去中心化但长且无序；Snowflake 趋势递增、性能好，但依赖时钟和机器号；号段模式吞吐高但要处理号段浪费和服务可用性。

选型要看是否需要趋势递增、可读性、短 ID、全局唯一、跨机房和高吞吐。

## 分布式锁怎么设计才安全？

参考回答：

Redis 锁至少要用 `SET key token NX PX ttl`，释放时用 Lua 校验 token 后删除，避免误删别人的锁。要考虑业务执行超过 TTL、续租、锁获取超时、锁服务不可用。

更重要的是不要把分布式锁当成事务。核心正确性还要靠数据库唯一约束、乐观锁、状态机和幂等。

## 幂等怎么设计？

参考回答：

幂等是同一个操作重复执行多次，结果和执行一次一致。常见方案包括 idempotency key、业务唯一约束、去重表、状态机、版本号、请求流水和消息消费记录。

写接口、支付、退款、发券、MQ 消费都要设计幂等。幂等最好落到持久化约束，不能只靠内存或短 TTL 缓存。

## 分布式事务有哪些方案？

参考回答：

常见方案包括 2PC、TCC、Saga、本地消息表、Outbox。2PC 一致性强但可用性和性能差；TCC 需要业务实现 Try/Confirm/Cancel；Saga 适合长流程补偿；Outbox 适合保证本地事务和消息发送一致。

多数互联网业务不追求全局强事务，而是通过状态机、事件驱动、补偿和对账实现最终一致。

## 如何做服务降级和熔断？

参考回答：

降级是在依赖不可用或系统压力大时，关闭非核心能力或返回兜底结果。熔断是在下游错误率或超时超过阈值时，短时间停止调用，避免级联故障。

核心交易不能假成功；非核心能力如推荐、画像、通知可以降级。熔断要配合半开探测、监控和恢复策略。

## 如何防止雪崩？

参考回答：

雪崩通常来自下游故障、缓存大面积失效、重试风暴或线程池被慢依赖拖死。治理要从隔离、限流、熔断、降级、超时、重试 backoff、缓存 TTL jitter 和容量保护入手。

面试里要强调：重试必须受控，否则故障时会把流量放大。
