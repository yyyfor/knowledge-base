# Java IO and Network Interview Answers

IO 和网络题要从阻塞模型、线程模型、连接管理、超时和内存使用角度回答。

## BIO、NIO、AIO 有什么区别？

参考回答：

BIO 是阻塞 IO，通常一个连接对应一个线程，模型简单，但高并发下线程资源消耗大。NIO 是非阻塞 IO，可以用 Selector 管理多个 Channel，减少一连接一线程的浪费。AIO 是异步 IO，由系统完成后回调通知应用，但在 Java 生态里服务端高并发网络编程更常见的是基于 NIO 的框架。

成熟表达：NIO 不是简单“更高级”，它降低线程资源成本，但提高编程复杂度，所以真实项目常用 Netty 封装事件循环、编解码、连接管理和背压。

## NIO 的 Buffer、Channel、Selector 分别负责什么？

参考回答：

Channel 表示数据通道，可以从中读写数据；Buffer 是读写数据的内存缓冲区；Selector 可以监听多个 Channel 的事件，比如连接、读、写，从而让一个线程管理多个连接。

回答时可以补：NIO 编程要处理半包、粘包、状态机、内存管理和事件循环，所以业务系统通常不直接裸写 NIO。

## Netty 为什么常用于高并发网络通信？

参考回答：

Netty 基于 NIO，封装了 Reactor 线程模型、ChannelPipeline、编解码、连接管理、ByteBuf 和异步事件处理。它让开发者更容易写高性能 TCP/RPC 服务。

但 Netty 不是银弹。要避免在 event loop 里执行阻塞操作，否则会拖慢同一个 event loop 上的所有连接。耗时业务逻辑应该放到业务线程池。

## 序列化和反序列化有什么风险？

参考回答：

序列化把对象转成可传输或存储的格式，反序列化再恢复对象。风险包括性能开销、版本兼容、字段演进、安全漏洞和不可信输入导致的反序列化攻击。

实际项目里要避免对不可信数据使用危险的原生反序列化，优先使用 JSON、Protobuf 等可控协议，并做好 schema 演进和字段兼容。

## HTTP 和 HTTPS 的区别是什么？

参考回答：

HTTP 是明文传输，HTTPS 在 HTTP 之下加入 TLS，提供加密、身份认证和完整性保护。TLS 通过证书验证服务器身份，协商会话密钥后用对称加密传输数据。

面试可以补：HTTPS 保护传输链路，但不等于接口安全。仍然需要认证授权、参数校验、防重放、防刷、日志脱敏和密钥管理。

## TCP 三次握手和四次挥手解决什么问题？

参考回答：

三次握手用于确认双方收发能力，并同步初始序列号。四次挥手用于双方分别关闭发送方向，因为 TCP 是全双工连接，一方关闭发送不代表另一方不能继续发送。

业务系统里更常见的连接问题是连接池耗尽、TIME_WAIT 过多、连接超时、读取超时和 keep-alive 配置不合理。

## 连接超时和读取超时有什么区别？

参考回答：

连接超时是建立连接阶段超时，通常说明网络不可达、服务端拒绝或建连太慢。读取超时是连接已经建立，但等待响应数据超时，通常说明下游处理慢、线程池满、数据库慢或网络抖动。

服务调用必须设置连接超时和读取超时。没有超时的外部调用会拖住业务线程，最终把 Tomcat 线程池或业务线程池耗尽。

## 长连接、连接池和 keep-alive 怎么理解？

参考回答：

长连接是复用同一 TCP 连接处理多个请求，减少频繁建连成本。连接池管理一组可复用连接，控制最大连接数、空闲时间和可用性。keep-alive 是维持连接复用的机制。

连接池不是越大越好。太小会排队，太大会压垮下游或本机资源。要结合下游容量、请求耗时、并发量和超时配置设置。

## 大文件上传如何避免内存打爆？

参考回答：

不要一次性把文件读入内存。应该用流式读取、分片上传、临时文件或对象存储直传，限制单文件大小和并发数。处理链路中要有 backpressure，避免上游上传速度远大于下游处理能力。

如果要做校验或解析，也应分块处理，并把耗时任务异步化。

## 如何处理接口偶发超时？

参考回答：

先看超时发生在哪一段：客户端到网关、网关到服务、服务内部线程池、数据库、缓存、外部 RPC。用 trace、日志、metrics 对齐同一个请求的耗时。再看是否有连接池耗尽、GC、锁竞争、慢 SQL、下游抖动或队列积压。

修复可以包括超时配置、连接池调整、慢查询优化、限流降级、隔离线程池、重试 backoff，但前提是保证幂等。
