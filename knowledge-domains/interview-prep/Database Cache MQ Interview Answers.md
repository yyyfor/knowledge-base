# Database Cache MQ Interview Answers

数据库、缓存、消息队列是后端面试最常见的三件套。核心不是“会不会用 MySQL/Redis/Kafka”，而是看你能不能解释一致性、性能、故障和数据正确性。

## 数据库事务 ACID 是什么？

参考回答：

ACID 包括原子性、一致性、隔离性和持久性。原子性保证事务内操作要么都成功要么都失败；一致性保证数据从一个合法状态到另一个合法状态；隔离性控制并发事务之间的可见性；持久性保证提交后的数据不会因为崩溃丢失。

面试里要补：ACID 不是免费的，隔离级别越强，并发性能通常越低。核心交易数据需要强事务，日志、统计、消息通知可以接受最终一致。

## 事务隔离级别如何选择？

参考回答：

常见隔离级别包括 Read Uncommitted、Read Committed、Repeatable Read、Serializable。隔离级别越高，越能避免脏读、不可重复读和幻读，但并发成本越高。

在 MySQL InnoDB 中，默认 Repeatable Read 通常适合大多数业务。但一些高并发更新场景仍要关注当前读、锁范围和索引命中。面试中最好结合扣库存、余额更新、订单状态机来讲。

## 索引为什么会失效？

参考回答：

常见原因包括不符合最左前缀、对索引列做函数或计算、隐式类型转换、前导模糊匹配、OR 条件不当、选择性太低、排序和过滤不匹配。

判断不能靠猜，要看 `EXPLAIN` 的 type、key、rows、Extra。索引设计要从查询模式出发，而不是看到字段就加索引。

## 覆盖索引和回表是什么？

参考回答：

覆盖索引是查询所需字段都能从二级索引里拿到，不需要再回到主键索引查完整行。回表是通过二级索引找到主键后，再去聚簇索引取其他列。

覆盖索引可以减少随机 I/O，提高查询性能。但索引字段太多会增加写入成本和存储成本，所以要服务于高频查询。

## 缓存和数据库如何保证一致性？

参考回答：

常见模式是 cache-aside：读时先查缓存，miss 再查 DB 并回填；写时先更新 DB，再删除缓存。删除缓存通常比更新缓存更稳，因为更新缓存可能受并发写顺序影响。

完全强一致很难且成本高。多数业务追求最终一致，通过延迟双删、消息通知、binlog 订阅、TTL 和重试来降低不一致窗口。核心交易真相仍应在数据库，缓存只是加速层。

## 缓存穿透、击穿、雪崩怎么处理？

参考回答：

缓存穿透是查不存在的数据，处理方式包括参数校验、null cache、Bloom filter。缓存击穿是热点 key 过期瞬间大量请求打到 DB，处理方式包括互斥锁、singleflight、逻辑过期和热点预热。缓存雪崩是大量 key 同时失效或 Redis 故障，处理方式包括 TTL jitter、多级缓存、限流降级和 Redis 高可用。

回答时要讲清三个词的区别，不要混成一个“加缓存”。

## Redis 和本地缓存怎么选？

参考回答：

本地缓存延迟低、不走网络，适合小规模、低变化、高频读的数据；缺点是多实例一致性差，占用应用内存。Redis 是集中式缓存，容量和一致性治理更好，支持丰富数据结构，但有网络开销和 Redis 可用性问题。

常见组合是本地缓存做一级缓存，Redis 做二级缓存。高风险数据不要只靠本地缓存，要有 TTL、版本号或主动失效机制。

## 消息队列解决什么问题？

参考回答：

消息队列主要解决削峰、解耦和异步。削峰是把突发流量缓冲下来；解耦是让生产者不依赖消费者实时可用；异步是把非核心耗时任务从主链路移走。

但 MQ 引入新问题：重复消费、消息丢失、顺序性、积压、延迟、DLQ、幂等和最终一致。面试里不能只讲好处。

## 如何保证消息不丢？

参考回答：

要看生产、Broker、消费三个阶段。生产端要确认发送结果，失败重试；Broker 端要持久化和副本；消费端要处理成功后再 ack。业务上最好用 Outbox 或本地消息表保证“业务写库”和“发消息”一致。

严格不丢通常意味着更高延迟和成本。面试中要说明业务要求：核心资金消息和普通通知消息的可靠性要求不同。

## 如何处理消息积压？

参考回答：

先判断积压原因：生产突然变多、消费者变慢、下游慢、单分区热点、消息处理失败反复重试。短期可以扩消费者、提高并发、临时跳过低优先级消息、拆分 topic/partition、降级下游。长期要做消费耗时监控、DLQ、限流、分区设计和容量评估。

不能只加消费者。如果瓶颈在数据库或下游服务，加消费者会把下游打挂。

## 如何保证消息顺序？

参考回答：

全局顺序成本很高，通常只保证某个业务 key 内有序，比如同一个订单的状态变更按 orderId 分区。Kafka/RocketMQ 这类系统可以通过同 key 进同 partition/queue 实现局部有序。

消费者也要单线程或按 key 串行处理，否则拉取有序但处理乱序。能不用强顺序就不要用，很多业务可以通过状态机和版本号容忍乱序。

## MQ 和定时任务怎么选？

参考回答：

MQ 适合事件驱动、异步解耦和近实时处理。定时任务适合周期性扫描、补偿、对账和低频批处理。

生产系统常两者结合：主流程用 MQ 推动，定时任务做兜底补偿。例如订单支付回调发消息，定时任务扫描长时间未完成订单做关闭或补偿。
