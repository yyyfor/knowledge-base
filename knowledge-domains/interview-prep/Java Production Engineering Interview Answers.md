# Java Production Engineering Interview Answers

Production Java 题看的是你是否真的跑过服务：超时、重试、幂等、限流、日志、监控、灰度、回滚、排障。

## 如何设计一个高可靠的下单接口？

参考回答：

先定义主链路：校验用户和商品、幂等检查、锁定库存、计算价格、创建订单、发起支付或等待支付、发送消息。核心是保证状态机清晰、重复请求不重复下单、库存不会超卖、失败可恢复。

工程上要有 idempotency key、唯一约束、事务边界、库存扣减策略、消息 outbox、超时和重试、监控告警。非核心动作如发短信、推荐更新可以异步化。

## 如何避免重复提交？

参考回答：

可以用前端按钮置灰，但真正可靠的幂等必须在后端。常见方案是请求幂等号、业务唯一键、数据库唯一约束、去重表、状态机校验或 Redis 短期 token。

面试里要强调：幂等不只是防用户连点，也要防网络重试、MQ 重复消费、客户端超时后再次提交。

## 如何做接口限流？

参考回答：

限流可以按用户、IP、租户、接口、资源或全局维度做。算法包括固定窗口、滑动窗口、漏桶、令牌桶。入口层可以做粗粒度限流，服务内部对关键资源做细粒度限流。

限流后要有明确响应和降级策略。核心交易不能假成功，非核心功能可以降级或异步处理。

## 如何处理下游服务超时？

参考回答：

所有下游调用都要有连接超时和读取超时。超时后要根据业务语义决定失败、降级、重试或异步补偿。不能无限等待，也不能无脑重试。

如果下游持续变慢，要用熔断、隔离线程池、限流和 fallback 防止级联故障。

## 如何设计重试？

参考回答：

只对可重试错误重试，比如临时网络错误、限流、短暂不可用。业务校验失败、参数错误、余额不足不应重试。重试要有最大次数、指数退避、jitter，并且必须保证幂等。

重试会放大流量，故障时大量重试可能压垮下游，所以要配合熔断和限流。

## 日志应该打什么？

参考回答：

日志要能帮助复盘一次请求：trace id、业务 id、用户/租户、关键状态变化、外部调用结果、异常栈、耗时和错误码。不要打印密码、token、身份证、银行卡等敏感信息。

日志不是越多越好。高频日志要采样或降级，避免 IO 压力和成本失控。

## 如何设计统一异常处理？

参考回答：

统一异常处理要把异常分层：参数错误、认证授权失败、业务异常、下游异常、系统异常。对用户返回稳定错误码和可理解信息，对内部保留 trace id 和详细日志。

不要把所有异常都吞掉返回成功，也不要把内部栈信息暴露给客户端。

## 如何做灰度发布和回滚？

参考回答：

灰度可以按用户、租户、地区、流量比例、版本或开关逐步放量。发布前要有监控指标、错误率、延迟、业务指标和回滚方案。发布后观察核心指标，再逐步扩大范围。

回滚不仅是代码回滚，还要考虑数据库 schema、配置、消息格式和缓存兼容。不可逆变更要拆成多阶段发布。

## 如何排查线上慢接口？

参考回答：

先用监控确认是平均慢还是 p99 慢，是单接口还是全站慢。再看 trace 拆分耗时：网关、应用、数据库、缓存、RPC、队列。应用内部看线程池、连接池、GC、锁竞争、慢 SQL、外部服务。

定位后再优化，不要一上来改代码。常见修复包括索引、批量查询、缓存、连接池调整、异步化、限流和下游隔离。

## 如何提升接口吞吐？

参考回答：

先找瓶颈：CPU、内存、锁、线程池、连接池、DB、Redis、下游 RPC、网络。然后针对瓶颈优化。常见手段包括减少重复计算、批处理、缓存、异步化、连接复用、SQL 优化、减少锁粒度、水平扩展。

吞吐提升不能牺牲正确性。交易接口要先保证幂等、一致性和可恢复，再谈性能。

## 成熟回答框架

- timeout：所有外部调用都要有超时。
- retry：只对可重试错误重试，并且要 backoff。
- idempotency：重试前先保证不会重复写。
- circuit breaker：下游异常时避免级联故障。
- observability：metrics、logs、trace、业务指标。
- fallback：非核心能力可以降级，核心交易不能假成功。
