# Java Security Interview Answers

安全题不要只讲算法或 token，要覆盖认证授权、权限模型、数据保护、输入校验、审计和风控。

## 认证和授权有什么区别？

参考回答：

认证是确认“你是谁”，比如登录、密码、短信验证码、OAuth。授权是确认“你能做什么”，比如某用户能不能访问某资源、执行某操作。

系统设计里要把两者拆开：先 authentication 得到可信身份，再 authorization 根据 subject、resource、action、condition 做权限判断。

## JWT 的优缺点是什么？

参考回答：

JWT 的优点是无状态、跨服务验证方便、可以携带有限 claims。缺点是签发后撤销困难，payload 只是 Base64URL 编码不是加密，token 过大也会增加请求开销。

成熟方案通常是短有效期 access token + refresh token，配合 token version、黑名单或设备会话管理来处理退出登录、改密和泄漏。

## Session 和 JWT 怎么选？

参考回答：

Session 把登录状态存在服务端，客户端只保存 session id，撤销和服务端控制更容易，但需要 session 存储和集群共享。JWT 把状态放在 token 中，服务端验证签名即可，适合多服务和跨域场景，但撤销和权限变更实时生效更难。

核心不是谁更先进，而是看撤销要求、系统拓扑、移动端支持、安全策略和运维复杂度。

## SSO 单点登录怎么实现？

参考回答：

SSO 的核心是统一身份提供方。用户访问业务系统时，如果未登录，跳转到身份中心认证；认证成功后身份中心签发 ticket/token，业务系统验证后建立自己的会话。

常见协议包括 OAuth2、OIDC、SAML。面试里要补安全点：redirect_uri 校验、state 防 CSRF、token 有效期、签名校验和权限同步。

## RBAC 和 ABAC 有什么区别？

参考回答：

RBAC 基于角色授权，比如用户属于财务经理角色，就拥有某些菜单和操作权限。ABAC 基于属性授权，比如用户部门、资源归属、金额、时间、地区、数据敏感级别共同决定是否允许。

RBAC 简单易治理，适合大多数后台系统；ABAC 更灵活，适合复杂数据权限和动态策略，但规则治理更难。大型系统常组合使用。

## 为什么前后端都要做参数校验？

参考回答：

前端校验提升用户体验，减少无效请求；后端校验保证安全和数据正确性。前端不可信，攻击者可以绕过页面直接调接口，所以所有关键校验必须在后端执行。

后端校验包括类型、长度、范围、枚举、权限、业务状态和幂等。参数校验失败要返回明确错误，不要把内部异常暴露给用户。

## 敏感数据如何脱敏？

参考回答：

敏感数据保护要覆盖传输、存储、展示和日志。传输用 TLS，存储可加密或 tokenization，展示做 mask，日志避免打印明文身份证、银行卡、手机号、token、密码。

脱敏还要考虑权限：不是所有内部员工都应该看到完整数据。对高敏操作要有审批、审计日志和异常访问监控。

## 密码为什么只能重置，不能找回？

参考回答：

系统不应该保存明文密码，也不应该可逆加密保存密码。正确方式是使用强哈希算法加盐存储，比如 bcrypt、scrypt、Argon2。用户忘记密码时只能通过身份验证后重置新密码。

如果系统能“找回原密码”，说明密码存储设计有严重安全问题。

## 常见加密算法怎么分类？

参考回答：

常见分类包括哈希、对称加密、非对称加密和签名。哈希用于摘要和密码存储；对称加密如 AES，适合大量数据加密；非对称加密如 RSA/ECC，适合密钥交换和身份验证；签名用于证明数据来源和完整性。

回答时要补密钥管理：加密强度不仅取决于算法，还取决于密钥生成、存储、轮换、权限和审计。

## 如何防止接口被刷？

参考回答：

可以从身份、频率、资源和风险四层防护：登录态、API key、设备指纹、IP/用户/租户限流、验证码、风控规则、黑名单、成本型挑战、异步审核。

实现上要区分普通限流和安全风控。普通限流保护系统容量，风控识别异常行为。核心接口还要有幂等、防重放和审计。
