# Java Distributed Systems Interview Answers

分布式题考的是故障视角：网络会抖、服务会慢、消息会重复、数据会不一致、节点会宕机。

## CAP 和 BASE 怎么理解？

参考回答：

CAP 讨论的是分布式系统在网络分区发生时，一致性和可用性之间的取舍。真实系统必须容忍分区，所以要选择在故障时更偏一致性还是更偏可用性。BASE 是对强一致的一种工程妥协：基本可用、软状态、最终一致。

面试里不要只背定义，要落到场景：支付扣款更偏一致，信息流点赞数可以最终一致。

## RPC 和 HTTP API 有什么区别？

参考回答：

HTTP API 更通用、跨语言和调试友好，适合开放接口和资源模型。RPC 更像本地方法调用，通常有强类型 IDL、代码生成、连接复用和更高性能，适合内部服务间调用。

选择取决于调用方范围、性能要求、治理能力、协议兼容和团队生态。内部高频调用可用 gRPC/Dubbo，对外开放接口更常用 REST/HTTP。

## Dubbo / gRPC 这类 RPC 框架解决什么问题？

参考回答：

RPC 框架通常解决服务发现、负载均衡、序列化、连接管理、超时、重试、熔断、限流和观测。它们把远程调用封装成接近本地方法调用的体验。

但远程调用永远不是本地调用。必须设置超时、处理失败、避免无限重试，并设计幂等。

## 分布式 ID 有哪些方案？

参考回答：

常见方案包括数据库自增、号段模式、UUID、Snowflake、Redis/数据库计数器。数据库自增简单但扩展性和可用性有限；UUID 去中心化但长且无序；Snowflake 趋势递增、性能好，但依赖机器号和时钟；号段模式性能高但要处理号段预取和浪费。

选型看是否需要全局唯一、趋势递增、可读、短 ID、高吞吐和跨机房。

## 一致性哈希解决什么问题？

参考回答：

一致性哈希用于在节点增减时减少 key 迁移范围。普通取模在节点数变化时大量 key 会重新映射；一致性哈希把节点和 key 映射到环上，节点变化只影响相邻区间。

为了避免数据倾斜，通常会引入虚拟节点。

## 分布式锁有哪些实现？

参考回答：

常见实现有 Redis、ZooKeeper、数据库唯一约束或行锁。Redis 锁性能高，但要处理 token、过期、续租和释放校验；ZooKeeper 基于临时顺序节点，语义更强但性能和运维成本更高；数据库锁简单但吞吐有限。

更重要的是先问是否真的需要锁。有些场景用唯一约束、状态机、乐观锁或幂等 key 更可靠。

## 分布式事务怎么处理？

参考回答：

方案包括 2PC、TCC、Saga、本地消息表、Outbox 和最终一致。2PC 一致性强但阻塞和可用性差；TCC 需要业务实现 Try/Confirm/Cancel；Saga 把长事务拆成多个本地事务和补偿；Outbox 把业务写入和消息写入放在同一个本地事务里，再异步投递。

互联网业务里很多场景更适合最终一致，而不是强行做全局事务。

## 本地消息表、Outbox、Saga、TCC 有什么区别？

参考回答：

本地消息表和 Outbox 关注“业务状态变更和消息发送一致”，通过本地事务先落库，再异步投递。Saga 关注跨服务长流程，每一步都有补偿动作。TCC 关注资源预留和确认/取消，业务侵入性更强。

选型看业务是否能补偿、是否需要资源预留、是否能接受最终一致、失败后人工介入成本多高。

## 配置中心解决什么问题？

参考回答：

配置中心集中管理应用配置，支持动态变更、环境隔离、灰度发布、权限控制和审计。它避免配置散落在机器或代码里。

但动态配置要谨慎，尤其是线程池、限流、开关和金额规则。需要变更审计、回滚能力和配置校验，避免误配置造成事故。

## API Gateway 的职责是什么？

参考回答：

API Gateway 位于入口层，常见职责包括路由、认证、鉴权、限流、熔断、灰度、协议转换、日志和 trace 注入。它不应该承载过多业务逻辑，否则会变成难维护的中心化业务层。

面试里可以说：网关负责通用入口治理，业务规则仍应下沉到对应服务。
